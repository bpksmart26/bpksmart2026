**작성일**: 2026-05-08

**상태**: 설계 단계 (구현 전)
**관련 파일**: `apps_script/Code.gs`, `공급기업_관리.html`, `config.js`

## 1. 개요

### 1.1 최종 목적

**관리자는 견적 웹페이지와 노션 페이지 두 곳만 보면서 운영한다.** 스프레드시트는 raw 저장소·운영 백업 용도로만 둔다.

- 신청 데이터와 견적 데이터를 **한 고객 = 한 행 / 한 노션 페이지**로 묶어서 관리한다.
- 시트와 노션은 양방향 동기화. 단 양방향 범위는 운영 필드에 한정한다.

### 1.2 비범위 (Out of Scope)

본 설계는 다음을 다루지 않는다.

- 스프레드시트 헤더 한글화 (별도 결정으로 보류 — 라벨 패치 미적용 결정 완료).
- Notion에서 시트로의 본 데이터(회사명·연락처·견적 항목 등) 수정 반영. 단방향 컬럼은 노션에서 수정해도 시트값으로 되돌린다.
- Notion에서 신규 페이지(시트에 대응 없는)를 만들었을 때의 시트 역방향 생성. sync는 시트가 만든 페이지(`신청ID` 속성을 가진 페이지)만 관리한다.
- 노션 페이지 본문 영역의 동기화. 본문은 운영자 자유 영역으로 보존한다.

### 1.3 결정 사항 요약

| 결정 | 채택 옵션 |
|---|---|
| 노션 연결 방식 | A — Apps Script가 Notion REST API 직접 호출 (`UrlFetchApp`) |
| 양방향 동기화 범위 | 옵션 1 — 운영 필드 4개만 양방향, 본 데이터는 단방향 |
| 단방향 정책 | B — "되돌리기" (노션에서 수정 시 다음 sync에서 시트값으로 덮어쓰기) |
| 노션 DB 스키마 | C — 사용자가 빈 DB + GAS의 `ensureNotionSchema`가 빠진 컬럼 자동 보충 |
| 한글/영어 매핑 | 코드 안에 `NOTION_PROP_MAP` 매핑 테이블 단일 출처 |
| 폴링 주기 | 5분 (GAS Time Trigger) + 운영자용 즉시 동기화 버튼 |
| 기존 데이터 마이그레이션 | 불필요 (시트 데이터 없음) |
| 시트 삭제 시 노션 | A2 — Notion archive (휴지통, 30일 보관) |
| 노션 삭제 시 시트 | B1 — 무시, 다음 sync에서 자동 재생성 |
| 같은 사업자 재신청 | C1 — 기존 노션 페이지 갱신 (page ID 유지, 운영자 메모 보존) |

### 2.3 노션 운영자 자유 추가 컬럼 (sync 무관)

운영자가 노션 DB에 자유롭게 추가 가능하며, sync는 절대 건드리지 않는다.

예: `우선순위 (Select)`, `다음 컨택일 (Date)`, `내부 코멘트 (Text)`, `태그 (Multi-select)`, `담당팀 (Select)`, `진행도 % (Number)`.

## 3. 시스템 아키텍처

### 3.1 컴포넌트 다이어그램

```
[고객 신청 폼]                [공급기업_관리 (관리자)]
신청기업_장비신청.html         공급기업_관리.html
        │                              │
        │ saveApp                       │ saveQt / updateApp / deleteApps
        │                              │ syncNotionNow (사이드바 버튼)
        ▼                              ▼
        ┌──────────────────────────────────────┐
        │  Apps Script (Code.gs) — 동기화 허브   │
        │                                      │
        │  • doPost (라우터)                    │
        │  • upsertUnified (시트 통합)         │
        │  • pushToNotion (시트→노션)          │
        │  • syncFromNotion (노션→시트, 폴링)  │
        │  • archiveNotionPage (삭제 처리)     │
        │  • _sync_queue retry                 │
        └─────┬────────────────────┬───────────┘
              │                    │
       즉시 (<3s)             5분 폴링 + 즉시 버튼
              │                    │
              ▼                    ▼
     ┌─────────────────┐   ┌─────────────────┐
     │ Spreadsheet     │   │ Notion DB       │
     │ ├─ 장비          │   │ (운영자 화면)    │
     │ ├─ 신청          │   └─────────────────┘
     │ ├─ 견적          │
     │ ├─ 통합정보 NEW │
     │ ├─ 설정          │
     │ └─ _sync_queue NEW │
     └─────────────────┘
```

### 3.2 데이터 흐름 — 시나리오별

#### 시나리오 A — 새 신청 들어옴

```
1. 고객 신청 폼 → apiCall('saveApp', app)
2. doPost: appendRow(SN.APP, ...) — 신청 시트에 raw row
3. doPost: upsertUnified(app) — 통합정보 upsert (bizno 매칭)
4. doPost: pushToNotion(unifiedRow) — 노션 페이지 upsert (신청ID 매칭)
5. 모든 단계 try-catch로 격리: Notion 실패해도 시트 저장은 성공
6. 응답 반환 (1~3초)
```

#### 시나리오 B — 견적 발급

```
1. 관리자 견적 작성 → apiCall('saveQt', quote)
2. doPost: saveQuoteWithVersion — 기존 동작 (version/isLatest 관리)
3. doPost: upsertUnified(app, quote) — 통합정보의 해당 신청 row에 견적 컬럼 채움
4. doPost: pushToNotion(unifiedRow) — 노션 페이지 PATCH (견적 컬럼만)
5. 응답 반환
```

#### 시나리오 C — 같은 사업자 재신청

```
1. saveApp (시나리오 A와 동일 진입)
2. upsertUnified:
   - bizno로 통합정보 시트 검색
   - 기존 row 발견 → row 자체를 새 신청 데이터로 덮어쓰기
   - 견적 컬럼은 비움 (새 신청엔 아직 견적 없음)
   - id 컬럼도 새 신청 id로 갱신
3. pushToNotion:
   - **매칭 키는 사업자번호 (`사업자번호 = bizno` 로 query)** — 신청ID는 재신청 시 바뀌므로 안정적 매칭 키가 아님
   - 매칭되는 페이지 발견 → PATCH로 갱신 (page ID 유지)
   - 운영자가 그 페이지에 적은 메모/하위페이지/태그 모두 보존
   - `신청ID` 속성도 새 신청 id로 갱신 (디스플레이용)
   - 매칭 페이지 없음 → POST로 새 페이지 생성
```

#### 시나리오 D — 운영자가 노션에서 양방향 필드 수정

```
1. (5분 후) GAS Time Trigger → syncFromNotion()
2. Notion API: query DB where last_edited_time > LAST_SYNC_AT
3. 변경된 페이지마다:
   a. 양방향 4개 필드(status/manager/memo/quoteMemo)만 추출
   b. 신청 시트의 해당 row 업데이트
   c. 견적 시트의 해당 row 업데이트 (quoteMemo만)
   d. 통합정보 시트도 업데이트
   e. 변경된 필드 해시를 기록 (다음 pushToNotion이 동일 변경을 다시 push 안 하도록 — 무한루프 방지)
4. LAST_SYNC_AT 갱신
```

#### 시나리오 E — 신청 삭제

```
1. 관리자가 "중복 정리" 모달에서 신청 삭제 → apiCall('deleteApps', {ids})
2. doPost: 기존 deleteApps (신청 시트 row 삭제)
3. doPost: 영향받은 사업자번호별로 reconcile:
   - 잔여 신청 row가 있음 → 가장 최근으로 upsertUnified + pushToNotion (노션 페이지 갱신)
   - 잔여 신청 row가 없음 → 통합정보 row 삭제 + archiveNotionPage(bizno) (노션 archive)
4. archive는 사업자번호로 노션 페이지를 query해서 archived=true PATCH
5. 노션 휴지통 보관 (30일), 운영자가 복원 가능
```

> **중요**: 단순히 "삭제된 신청 ID에 대응하는 노션 페이지를 archive" 하면 안 됨. 같은 사업자가 신청 5건 보유 중 옛 1건만 삭제했을 때 노션 페이지를 archive 하면 운영자 데이터가 사라짐. **사업자 단위로 reconcile**해서 잔여 여부 판단 후 결정.

#### 시나리오 F — 즉시 동기화 버튼

```
1. 관리자가 사이드바 "🔄 노션 동기화" 클릭
2. 공급기업_관리.html → apiCall('syncNotionNow')
3. doPost: syncFromNotion() 즉시 실행
4. 응답: { ok:true, changed: n }
5. 토스트: "n건 변경 반영됨" (n=0이면 "변경 없음")
```

### 3.3 무한루프 방지

양방향 필드의 sync loop를 막는 방법:

1. **변경 감지 해시**: `pushToNotion` 호출 시, 양방향 4개 필드의 값을 hash로 기록 (`_lastPushedHash` Script Properties).
2. `syncFromNotion`이 노션→시트 갱신 후, 시트 변경이 다시 `pushToNotion` 트리거하지 않도록:
   - `pushToNotion`은 양방향 필드의 현재 값 hash가 `_lastPushedHash`와 동일하면 스킵.
3. 보조: `syncFromNotion`이 시트 갱신 후 `_lastPushedHash`를 새 값으로 직접 갱신.

### 3.4 충돌 해결

양방향 필드(`status` / `manager` / `memo` / `quoteMemo`)에서 시트와 노션이 거의 동시에 변경된 경우:

**Last-write-wins**:
- 시트 row의 변경 시각 (있으면) vs 노션 `last_edited_time`
- 더 최근 쪽이 source. 나머지를 덮어씀.
- 동시각이면 시트 우선 (raw 데이터 무결성).

5분 폴링 + 양방향 4필드 + 운영자 1~2명이라 충돌 빈도 매우 낮음. 단순 정책으로 충분.

### 3.5 오류 처리 (3-tier)

| 오류 | 영향 | 대응 |
|---|---|---|
| Notion 502/503 | 일시 실패 | retry × 2 (지수 백오프). 그래도 실패면 `_sync_queue` 적재 |
| Notion rate limit (3 req/s) | 일부 거부 | retry-after 헤더 따라 대기 + retry |
| Notion 401/403 (토큰 문제) | 모든 노션 호출 실패 | 시트 저장은 성공. 운영자에게 toast + GAS 로그 |
| 사용자가 노션 컬럼 삭제 | 해당 컬럼 PATCH 실패 | `ensureNotionSchema`가 자동 재생성 |
| 통합정보 시트 손상 | sync 깨짐 | `autoInitSheets`가 자동 재생성. 데이터는 `rebuildUnified()` 수동 함수로 신청+견적 시트로부터 일괄 복구 |

#### `_sync_queue` 시트

| 컬럼 | 용도 |
|---|---|
| id | 큐 항목 ID |
| action | `pushToNotion` / `archiveNotionPage` 등 |
| payload_json | 원본 호출 인자 |
| retry_count | 재시도 횟수 |
| last_error | 마지막 에러 메시지 |
| created_at | 큐 적재 시각 |

매 폴링 시 큐 확인 → 재처리. retry_count 3회 초과 시 운영자 알림 (toast + 시트에 마킹).

## 4. 컴포넌트 (구현 단위)

### 4.1 `apps_script/Code.gs` 신규 함수 (8개)

| 함수 | 역할 | 호출 시점 |
|---|---|---|
| `getNotionConfig()` | Script Properties에서 `NOTION_TOKEN`, `NOTION_DB_ID` 읽기 | 모든 노션 호출 시작 |
| `notionFetch(method, path, payload)` | UrlFetchApp 래퍼: 인증 헤더, JSON 직렬화, retry | 모든 노션 호출 |
| `ensureNotionSchema()` | DB의 빠진 속성 자동 추가 (옵션 C) | sync 시작 시 1회/세션 |
| `upsertUnified(app, quote?)` | 통합정보 시트 row upsert (bizno 매칭) | saveApp / saveQt / updateApp / updateQt 후 |
| `pushToNotion(unifiedRow)` | 통합정보 row → 노션 페이지로 변환 + PATCH/POST. **사업자번호로 매칭** (재신청 시 신청ID 변경되므로 안정 키는 bizno) | upsertUnified 직후 |
| `formatItemsForNotion(items, options)` | items/options JSON → 사람 읽기 텍스트 (`견적요약`) | pushToNotion 내부 |
| `archiveNotionPage(bizno)` | 사업자번호로 노션 페이지 query → archived=true PATCH | deleteApps 후 reconcile에서 사업자의 잔여 신청이 0건인 경우 |
| `syncFromNotion()` | 노션 → 시트 폴링 (양방향 4필드) | 5분 Time Trigger + 즉시 버튼 |

### 4.2 `apps_script/Code.gs` 신규 보조 함수 (3개)

| 함수 | 역할 |
|---|---|
| `rebuildUnified()` | 신청+견적 시트로부터 통합정보 시트 일괄 재구축 (장애 복구용 수동 실행) |
| `_enqueueSync(action, payload, error)` | `_sync_queue` 시트에 적재 |
| `_processSyncQueue()` | 큐 재처리, syncFromNotion 안에서 호출 |

### 4.3 `apps_script/Code.gs` 기존 함수 수정 (3곳)

| 위치 | 수정 내용 |
|---|---|
| 상수 정의 영역 | `UNIFIED_COLS`, `UNIFIED_ARR`, `NOTION_PROP_MAP`, `NOTION_PROP_TYPES`, `BIDIRECTIONAL_FIELDS = ['status','manager','memo','quoteMemo']` 추가 |
| `autoInitSheets` | 통합정보 시트 + `_sync_queue` 시트 자동 생성 추가 |
| `doPost` | `case 'saveApp'`, `case 'updateApp'`, `case 'saveQt'`, `case 'updateQt'`, `case 'deleteApps'` 끝에 `upsertUnified` + `pushToNotion` (또는 `archiveNotionPage`) 호출 추가. 모두 try-catch 격리. 새 액션 `case 'syncNotionNow'` 추가 |

### 4.4 GAS Time Trigger (1개)

   - URL에서 DB ID 32자 hex 복사

3. **GAS Script Properties 등록** (1분)
   - Apps Script 에디터 → Project Settings → Script Properties
   - `NOTION_TOKEN`, `NOTION_DB_ID` 추가

4. **노션 가이드 callout 작성** (선택, 5~10분)

## 8. 미해결·향후 과제

### 8.1 향후 개선 후보 (현재 범위 외)

- Notion Webhooks (베타) 정식 지원 시 5분 폴링 → webhook 즉시 반영으로 전환
- 양방향 범위 옵션 1 → 옵션 2 확장 (일부 텍스트 필드 양방향 추가)
- 통합정보 시트의 외부 노출 (대시보드 차트 직접 연동)
- 노션 페이지 본문에 견적 PDF 임베드 자동화

### 8.2 운영 중 모니터링 포인트

- GAS UrlFetch 일일 한도 (무료 20,000회) — 5분 폴링 + push까지 합쳐 일 ~500회 수준 예상, 여유 큼
- Notion API rate limit (3 req/sec) — 쇄도 시 retry 큐가 흡수
- `_sync_queue` 누적 — 20행 초과 시 운영자 알림 (자동 해소 안 되는 구조적 문제 신호)

### 8.3 운영 가이드 (별도 문서)

- `docs/notion-sync/노션DB속성목록.md` — 39개 속성 카탈로그
- `docs/notion-sync/사용자작업체크리스트.md` — STAGE 1/2/3 단계별 체크리스트
- 운영 시작 후 `docs/notion-sync/운영가이드.md` 추가 작성 권장 (양방향 필드 사용법, 트러블슈팅)

## 9. 보안 고려

- **Notion Token은 Script Properties에만 저장**. 코드, 시트, git, 노션 페이지에 절대 노출 금지.
- Token 유출 의심 시: Notion Integration 페이지에서 Regenerate → 새 token을 Script Properties에 갱신.
- `config.js`의 Apps Script URL은 공개되어도 Notion Token이 분리되어 있어 안전.
- GAS deploy 시 access는 "Anyone with the link" 유지 (현 운영 방식과 동일). 인증은 `checkAuth`에 의존.
